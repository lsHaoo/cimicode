# 老版本代码迁移说明

## 迁移背景

老版本 opencode 基于较早的代码架构，定制代码直接混写在 server.ts 和各业务模块中。新版 opencode 进行了大规模重构（Effect 框架迁移、HttpApi 迁移、目录结构重组等），定制代码需要剥离到 `cimi-custom/` 目录下统一管理。

## 已完成的迁移

### 1. 路径修复（所有 cimi-custom 文件）

老版本使用相对路径，新版本目录结构变化后路径失效。已统一改用 `@/` 路径别名。

| 老版本路径 | 新版本路径 | 原因 |
|-----------|-----------|------|
| `"../util/log"` | `"@/util"` | Log 从 barrel re-export，`@/util/log` 不再独立导出 |
| `"../../config/config"` | `"@/config"` | Config 命名空间从 `@/config` barrel 导出 |
| `"../../skill"` | `"@/skill"` | 同上，使用 `@/` 别名避免层级依赖 |
| `"../../project/instance"` | `"@/project/instance"` | 同上 |
| `"../../util/lazy"` | `"@/util/lazy"` | 同上 |
| `"@opencode-ai/util/error"` | `"@opencode-ai/core/util/error"` | NamedError 迁移到 core 包 |
| `"@/global"` | `"@opencode-ai/core/global"` | Global 迁移到 core 包 |
| `"@/permission/next"` | `"@/permission"` | PermissionNext 不存在，统一用 `Permission` |

### 2. Effect 框架适配

新版 Config、Skill 等核心服务从普通模块变成了 Effect Service，访问方式完全不同：

```typescript
// 老版本（直接调用）
const config = await Config.get()
const skillInfo = await Skill.get(name)

// 新版本（通过 AppRuntime + Service.use）
const config = await AppRuntime.runPromise(Config.Service.use((cfg) => cfg.get()))
const skillInfo = await AppRuntime.runPromise(Skill.Service.use((svc) => svc.get(name)))
```

涉及文件：
- `skill-manager/status.ts` — Config.get() → AppRuntime + Config.Service
- `skill-manager/enable.ts` — Config.get() / Config.updateGlobal() → AppRuntime
- `skill-manager/disable.ts` — 同上
- `file-manager/upload.ts` — Config.get() → AppRuntime + Config.Service

### 3. 权限系统适配

```typescript
// 老版本
import { PermissionNext } from "@/permission/next"
const ruleset = PermissionNext.fromConfig({ skill: config.permission.skill })
const rule = PermissionNext.evaluate("skill", skillName, ruleset)

// 新版本
import { Permission } from "@/permission"
const ruleset = Permission.fromConfig({ skill: config.permission.skill })
const rule = Permission.evaluate("skill", skillName, ruleset)
```

### 4. 路由挂载方式

老版本：直接在 server.ts 里注册路由
```typescript
// 老版 server.ts
.route("/file-manager", FileManagerRoutes())
.route("/skill-manager", SkillManagerRoutes())
.route("/api/agi", AgiProxyRoutes())
.all("/*", proxyHandler)  // WebUI 代理
```

新版本：通过 adapter 层集成，不改 server.ts
```typescript
// mount-cimi-custom-routes.ts
export function mountCimiCustomRoutes(app: Hono) {
  app.route("/file-manager", FileManagerRoutes())
  app.route("/skill-manager", SkillManagerRoutes())
  // ...
}
// 在 adapter.bun.ts / adapter.node.ts 中调用
```

### 5. 废弃功能清理

- **toggle 路由**（`/skill-manager/toggle/:name`）已废弃，移除了路由定义和相关导出

### 6. 环境变量重命名

- `OPENCODE_WEB_URL` → `CIMICODE_WEB_URL`（避免与上游变量冲突）

### 7. 文件管理模块完整迁移

- `file-manager/delete.ts` — 已迁移，修复 import 路径
- `file-manager/download-folder.ts` — 已合并到 `download.ts`，原文件注释保留

## 已知限制

### download-folder.ts 已合并到 download.ts

`/download` 和 `/download-folder` 两个接口已合并为统一的 `/download` 接口，后端根据路径自动判断返回文件流还是 zip。`download-folder.ts` 原代码已注释保留，`getFolderSize` 的 `du -sb` Linux 依赖不变。

### restart.ts 的 Bun 依赖

`Bun.spawn` 仅在 Bun 运行时中可用。此接口仅用于 Web 端部署场景，桌面端（Electron / Node adapter）不使用此接口。try-catch 已兜底处理。

## 后续迁移注意事项

### 1. Effect 服务调用模式

新版本核心模块（Config、Skill、Permission、Auth、Instance 等）全部基于 Effect 框架。在 Hono 路由处理函数中访问这些服务必须通过：

```typescript
import { AppRuntime } from "@/effect/app-runtime"

// 读
const value = await AppRuntime.runPromise(Service.use((svc) => svc.method()))

// 写
await AppRuntime.runPromise(Service.use((svc) => svc.update(data)))
```

不要尝试直接调用服务方法（如 `Config.get()`），它们返回的是 Effect 而非 Promise。

### 2. 模块路径变更

迁移前务必确认目标模块在新版中的实际位置：

- `@opencode-ai/core/` — 从 opencode 包拆到 core 包的模块（Global、NamedError、Filesystem 等）
- `@/util` — Log 不再从 `@/util/log` 独立导出，从 barrel `@/util` 导入
- `@/config` — Config 命名空间从 `@/config`（index.ts barrel）导入，不是 `@/config/config`

### 3. Zod vs Effect Schema

新版本正在从 Zod 迁移到 Effect Schema，但目前 cimi-custom 中的路由验证仍使用 Zod（通过 `hono-openapi` 的 `validator`），这暂时没有问题。如果上游完全切换到 Effect Schema，后续可能需要适配。

### 4. Bun API 依赖

部分 cimi-custom 代码依赖 Bun 运行时 API：
- `Bun.spawn`（restart.ts）— 仅在 Bun 运行时可用
- `Bun.file`、`Bun.serve` 等 — 独立二进制内嵌了 Bun 运行时，可以正常使用

### 5. 老版本对比方法

对比老版本 server.ts 中的定制内容时，关注以下特征：
- 直接 import 了非标准路由（如 `FileManagerRoutes`、`AgiProxyRoutes`）
- `Flag.OPENCODE_WEB_URL` 相关逻辑
- `proxy` from `hono/proxy` 的使用
- 路由链中 `.route("/xxx", XxxRoutes())` 的非标准路由

### 6. 编译和构建

定制代码通过 `@/` 路径别名引用 opencode 内部模块，`bun build` 和 `Bun.build()` 都能正确解析。不需要额外的构建配置。
